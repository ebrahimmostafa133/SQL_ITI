1. Database Types Overview

1.1 Relational (SQL) Databases

Relational databases store data in tables (rows and columns). They were the dominant model for decades and excel at:

  • Transactional operations (ACID): Banking transactions, batch processing — situations where a set of steps must either all succeed or all fail.
  • Structured queries: Using SQL (Structured Query Language) to query, filter, join, and aggregate data.
  • Data integrity: Enforcing relationships and rules at the database level.

How data is stored:

ID Name Salary
1 Ahmed 50,000
2 Sara 60,000

1.2 NoSQL – Document-Based Databases

NoSQL databases (e.g., MongoDB) store data as documents (JSON-like objects). They were created to address limitations of relational databases.

Why NoSQL appeared:

  • In object-oriented backends (Java, Python, Node.js, etc.), data lives as objects/classes in memory.
  • Storing objects in a relational table requires ORM mapping (Object-Relational Mapping) — a translation layer.
  • For complex, nested data (e.g., a LinkedIn profile with arrays of jobs, education, contacts), relational databases require splitting data across multiple tables and JOINs to retrieve it — which is expensive.

LinkedIn Profile Example — Relational Approach:

users          → id, name, about
user_jobs      → user_id, title, company, start_date, description
user_education → user_id, degree, institution, year
user_contacts  → user_id, email, phone

To retrieve a full profile → must JOIN all tables → expensive I/O (disk seeks across multiple locations).

LinkedIn Profile Example — Document (NoSQL) Approach:

{
  "id": 1,
  "name": "Ahmed",
  "about": "...",
  "jobs": [{ "title": "Engineer", "company": "X", "start_date": "2020-01" }],
  "education": [{ "degree": "BSc", "institution": "Cairo University" }],
  "contacts": { "email": "a@b.com", "phone": "01..." }
}

The entire document is serialized (converted to a single string) and stored contiguously on disk. Retrieval is fast because everything is in one place.

Schema Flexibility: In NoSQL, one user can have 3 jobs and another can have 0 — no need to alter the schema.

1.3 Other Database Models

Model Description
Hierarchical Stores data in a tree structure (parent–child). Very old model.
Network Similar to hierarchical but allows multiple parent relationships.
Relational Tables with rows and columns. Most common.
Document-Based JSON-like documents (e.g., MongoDB).

1.4 The ORM Bridge

ORM (Object-Relational Mapping) solves the impedance mismatch between:

  • Backend: Data stored as objects with properties.
  • Database: Data stored as rows and columns.

An ORM maps each class to a table, and each property to a column. Examples: Hibernate (Java), SQLAlchemy (Python), Sequelize (Node.js), Entity Framework (.NET).

1.5 Trade-offs: Relational vs. NoSQL

Aspect Relational Document (NoSQL)
Data Retrieval (full object) Slow — requires JOINs across multiple tables Fast — data stored together
Data Update (single field) Fast — update only one column Slow — must rewrite entire document (re-serialize)
Schema flexibility Rigid — ALTER TABLE to add columns Flexible — each document can differ
Complex queries / analytics Excellent (SQL) Limited
Transactions (ACID) Strong Varies

⭐ What to Remember: NoSQL document stores are faster for reads (retrieve full object at once) but slower for updates (must rewrite the entire serialized document). Relational DBs are faster for partial updates (update one column, done).

2. MySQL Data Types

2.1 Numeric Types

Type Storage Range (Signed) Use Case
TINYINT 1 byte -128 to 127 Flags, small counts
SMALLINT 2 bytes -32,768 to 32,767 Small numbers
MEDIUMINT 3 bytes ~-8M to ~8M Medium integers
INT / INTEGER 4 bytes ~-2.1B to ~2.1B Standard IDs, counts
BIGINT 8 bytes ~-9.2×10¹⁸ to ~9.2×10¹⁸ Large counters
FLOAT 4 bytes Approximate Decimal numbers (approximate)
DOUBLE 8 bytes More precise Scientific calculations
DECIMAL(p,s) Variable Exact Money, financials — EXACT

⭐ Use DECIMAL for monetary values — FLOAT and DOUBLE are approximate (floating-point errors).

2.2 String/Text Types

Type Description Max Size
CHAR(n) Fixed-length string — always uses n bytes 255 chars
VARCHAR(n) Variable-length string — uses only what's needed + 1–2 bytes for length 65,535 bytes
TEXT Long text, no prefix length needed 65,535 chars
MEDIUMTEXT Larger text ~16 MB
LONGTEXT Very large text ~4 GB

CHAR vs. VARCHAR:

  • CHAR(10) always occupies 10 bytes, even if value is "Hi" (2 chars) — pads with spaces.
  • VARCHAR(10) stores "Hi" in 3 bytes (2 chars + 1 for length prefix).
  • CHAR SELECT advantage: Because the size is always fixed, MySQL doesn't need to check the length prefix on each row → slightly faster reads.
  • PostgreSQL note: PostgreSQL explicitly states CHAR may be worse in performance than VARCHAR. Always check your DB documentation.

⭐ What to Remember: Use VARCHAR by default. CHAR can be faster for fixed-width values (e.g., country codes, gender codes) but wastes space for variable-length data.

2.3 Date and Time Types

Type What It Stores Range Timezone Handling
DATE Date only (YYYY-MM-DD) '1000-01-01' to '9999-12-31' None
TIME Time only (HH:MM:SS) '-838:59:59' to '838:59:59' None
DATETIME Date + Time '1000-01-01 00:00:00' to '9999-12-31 23:59:59' None — stores as-is
TIMESTAMP Date + Time '1970-01-01 00:00:01' to '2038-01-19 03:14:07' Auto-converts to/from session timezone

The 2038 Problem: TIMESTAMP is stored as a 32-bit Unix integer. Its maximum value is January 19, 2038. After that date, TIMESTAMP columns will fail to store values. Use DATETIME for dates beyond 2038.

Timezone Behavior:

  • TIMESTAMP: When you INSERT a value, MySQL converts it from your session timezone to UTC and stores it. When you SELECT, it converts back to your session timezone. Different users in different timezones will see different times — MySQL handles this automatically.
  • DATETIME: Stores exactly what you give it. No timezone conversion. All users see the same value regardless of their timezone. You (the developer) are responsible for timezone handling in your application.

Best Practice: Store all times in UTC (Timezone offset +00:00), then convert to the user's local timezone in the application layer.

-- Set session timezone (example)
SET time_zone = '+05:00';

-- Check stored vs. retrieved values across timezone changes

Date Format in MySQL:

-- MySQL date format is: YYYY-MM-DD
INSERT INTO table_name (date_col) VALUES ('2026-03-29');

2.4 ENUM Type

ENUM is a string column that can only hold one value from a predefined list.

Syntax:

CREATE TABLE products (
  name     VARCHAR(100),
  size     ENUM('XS', 'S', 'M', 'L', 'XL')
);

What ENUM does:

  • Restricts the column to only the defined values — any other value causes an error.
  • Also enforces the exact format of the value (e.g., must be 'XL', not 'xl' or 'Extra Large').
  • Automatically adds a CHECK constraint — you don't need to add one manually.

How ENUM is stored internally:

  • MySQL stores ENUM values as their integer index (0, 1, 2…), not as strings.
  • 'XS' → index 0, 'S' → index 1, 'M' → index 2, etc.

Advantage: Smaller storage (integer vs. string), better readability in schema.

Gotcha — Sorting:

-- Ordering by an ENUM column sorts by INDEX, not alphabetically
SELECT * FROM products ORDER BY size;
-- Result: XS (0), S (1), M (2), L (3), XL (4)
-- NOT alphabetical order!

⭐ What to Remember: When you ORDER BY an ENUM column, MySQL sorts by the internal integer index, not the displayed string value. This can produce unexpected ordering if your ENUM values aren't defined in the order you expect.

2.5 Generated Columns

A generated column automatically computes its value from an expression applied to other columns. You never insert into it manually — MySQL calculates it for you.

Use Case Example: Store an email, auto-extract the domain.

CREATE TABLE users (
  email   VARCHAR(100),
  domain  VARCHAR(100) AS (SUBSTRING_INDEX(email, '@', -1)) STORED
);

INSERT INTO users (email) VALUES ('ahmed@ita.com');
-- domain is auto-filled as 'ita.com'

The SUBSTRING_INDEX function:

SUBSTRING_INDEX(string, delimiter, count)
  - count > 0 → take from the LEFT up to the Nth delimiter
  - count < 0 → take from the RIGHT up to the Nth delimiter

SUBSTRING_INDEX('ahmed@ita.com', '@', -1) → 'ita.com'

Two types of generated columns:

Type Keyword Storage Recalculated
Virtual VIRTUAL (default) NOT stored on disk Every time it's selected
Stored STORED Stored on disk Only once, when inserted/updated

Virtual vs. Stored — Trade-offs:

Aspect VIRTUAL STORED
Disk space No extra space Uses extra space
Read speed Slower (recalculates each time) Faster (reads from disk)
Write speed Faster (nothing to store) Slower (stores on write)
Best for Infrequent reads, simple expressions Frequent reads, complex expressions

Important Rule: The expression in a generated column must be deterministic — the same input must always produce the same output.

  • ✅ SUBSTRING_INDEX(email, '@', -1) → deterministic
  • ❌ NOW() → non-deterministic (returns different time each call)
  • ❌ RAND() → non-deterministic (returns random value)

You cannot manually insert into a generated column:

-- This will FAIL:
INSERT INTO users (email, domain) VALUES ('ahmed@ita.com', 'ita.com');
-- Error: domain is a generated column

3. Constraints

Constraints are rules applied to columns (or tables) that the database enforces before accepting any INSERT or UPDATE. They protect data integrity.

3.1 PRIMARY KEY 

CREATE TABLE employees (
  id    INT         PRIMARY KEY,
  name  VARCHAR(30) NOT NULL,
  email VARCHAR(100) UNIQUE NOT NULL
);
  • A PRIMARY KEY enforces two rules simultaneously:
    1. UNIQUE — no two rows may have the same value.
    2. NOT NULL — the value cannot be empty.
  • Only one PRIMARY KEY is allowed per table.
  • Every table should have a primary key.

3.2 NOT NULL 

name VARCHAR(30) NOT NULL
  • The column cannot be left empty (no NULL values).
  • A value must be provided on INSERT.

3.3 UNIQUE 

email VARCHAR(100) UNIQUE NOT NULL
  • The value must be unique across all rows.
  • UNIQUE alone (without NOT NULL) allows one NULL value — because NULL ≠ NULL in SQL.
  • UNIQUE + NOT NULL ≈ PRIMARY KEY, except a table can have multiple UNIQUE NOT NULL columns.

Difference: PRIMARY KEY vs. UNIQUE NOT NULL:

Feature PRIMARY KEY UNIQUE NOT NULL
Uniqueness ✅ ✅
NOT NULL enforced ✅ ✅ (explicitly)
Max per table 1 only Unlimited
Can have NULL? No No (you set NOT NULL)

3.4 CHECK Constraint 

salary INT CHECK (salary >= 30000)
  • Enforces a custom condition on the column value.
  • Any INSERT or UPDATE that violates the condition is rejected with an error.
-- Example: Enforce gender values
gender ENUM('Male', 'Female')
-- ENUM already acts as a CHECK constraint for allowed values

3.5 FOREIGN KEY

See Section 5 for full detail. A foreign key creates a relationship between two tables and enforces referential integrity — ensuring that a value in a child table actually exists in the parent table.

4. DML — Data Manipulation Language

4.1 INSERT

Insert all columns (must match order defined in CREATE TABLE):

INSERT INTO employees
VALUES (1, 'Ahmed', 'ahmed@email.com', NULL);

Insert specific columns (recommended):

INSERT INTO employees (id, name, email)
VALUES (1, 'Ahmed', 'ahmed@email.com');

Insert multiple rows at once:

INSERT INTO employees (id, name, email)
VALUES
  (1, 'Ahmed', 'ahmed@email.com'),
  (2, 'Sara',  'sara@email.com'),
  (3, 'Omar',  'omar@email.com');

Notes:

  • Strings go in single quotes 'value' or double quotes "value".
  • Dates go in quotes in 'YYYY-MM-DD' format.
  • If you don't specify column names, you must provide values for all columns in the correct order.
  • To skip a nullable column, either omit it (if columns are specified) or insert NULL.

4.2 UPDATE 

UPDATE employees
SET name = 'Mahmoud'
WHERE id = 1;

Update multiple columns:

UPDATE employees
SET name = 'Mahmoud', email = 'mahmoud@email.com'
WHERE id = 1;

Multiple conditions:

-- AND: both conditions must be true
UPDATE employees SET name = 'X' WHERE id = 1 AND name = 'Ahmed';

-- OR: at least one condition must be true
UPDATE employees SET department_id = 100 WHERE id = 1 OR id = 2;

⚠️ WARNING — UPDATE without WHERE:

UPDATE employees SET name = 'X';
-- This updates ALL rows in the table!

MySQL Workbench's Safe Update Mode will block UPDATE/DELETE without a WHERE clause using a primary key. To disable:

SET SQL_SAFE_UPDATES = 0;
-- Then run your UPDATE
-- Re-enable afterwards if needed
SET SQL_SAFE_UPDATES = 1;

4.3 DELETE 

DELETE FROM employees WHERE id = 1;

⚠️ WARNING — DELETE without WHERE:

DELETE FROM employees;
-- Deletes ALL rows!

DELETE syntax:

DELETE FROM table_name
WHERE condition;

5. Foreign Key Referential Actions

What Is a Foreign Key?

A Foreign Key (FK) creates a relationship between two tables. It ensures that any value in the child table's FK column must exist in the parent table's referenced column.

Terminology:

  • Parent table: The table that is referenced (e.g., departments).
  • Child table: The table holding the FK (e.g., employees).
  • Referential Integrity: The guarantee that no child record can point to a non-existent parent.

Example — Without FK (the problem):

-- Employee with department_id = 500, but no department 500 exists!
UPDATE employees SET department_id = 500 WHERE id = 2;
-- MySQL accepts this without complaint → Data integrity violated!

Adding a Foreign Key:

-- After tables are created, add the FK with ALTER TABLE:
ALTER TABLE employees
ADD CONSTRAINT fk_employee_department
FOREIGN KEY (department_id)
REFERENCES departments(id);

Or at table creation time:

CREATE TABLE employees (
  id            INT PRIMARY KEY,
  name          VARCHAR(30),
  department_id INT,
  CONSTRAINT fk_emp_dept FOREIGN KEY (department_id) REFERENCES departments(id)
);

Naming Convention: fk_<child_table>_<parent_table> (e.g., fk_employee_department).

5.1 RESTRICT (Default) 

FOREIGN KEY (department_id) REFERENCES departments(id)
ON DELETE RESTRICT
ON UPDATE RESTRICT
  • If a parent row has children referencing it, you cannot delete or update that parent row.
  • Error is thrown: MySQL refuses the operation.
  • This is the default if you don't specify any action.

Workaround to delete: First delete (or update) all child records, then delete the parent.

5.2 CASCADE 

ON DELETE CASCADE
ON UPDATE CASCADE
  • ON DELETE CASCADE: When you delete a parent row, all child rows referencing it are automatically deleted too.
  • ON UPDATE CASCADE: When you update the parent's PK, all child FK values are updated to the new value automatically.
-- Example: Delete department 100
DELETE FROM departments WHERE id = 100;
-- All employees in department 100 are also deleted!

5.3 SET NULL 

ON DELETE SET NULL
ON UPDATE SET NULL
  • When you delete a parent row, the child FK column is set to NULL.
  • The child row itself remains in the table — it just loses its parent reference.
  • Requirement: The FK column must allow NULL (no NOT NULL constraint).
-- Employee was in department 300
-- Department 300 is deleted
-- Employee's department_id becomes NULL (employee still exists)

5.4 Adding/Dropping a Foreign Key with ALTER TABLE

Drop an existing FK:

ALTER TABLE employees
DROP FOREIGN KEY fk_employee_department;

Re-create with different actions:

ALTER TABLE employees
ADD CONSTRAINT fk_employee_department
FOREIGN KEY (department_id)
REFERENCES departments(id)
ON DELETE CASCADE
ON UPDATE RESTRICT;

⭐ What to Remember: You cannot modify a FK in place. You must DROP it and recreate it.

Summary Table:

Action ON DELETE ON UPDATE
RESTRICT (default) Block deletion if children exist Block update if children exist
CASCADE Delete children automatically Update children's FK automatically
SET NULL Set children's FK to NULL Set children's FK to NULL

6. Subqueries (Nested Queries)

6.1 What Is a Subquery?

A subquery (or nested query / inner query) is a SELECT statement inside another SELECT, UPDATE, or DELETE statement.

When to use: When the information needed in a WHERE condition lives in a different table and you don't know the exact value upfront.

How it executes:

  1. The inner query (subquery) runs first.
  2. Its result is passed to the outer query.
  3. The outer query runs using that result.

Example — Find all employees in the "Open Source" department:

SELECT *
FROM employees
WHERE department_id = (
    SELECT id
    FROM departments
    WHERE name = 'Open Source'
);

The subquery SELECT id FROM departments WHERE name = 'Open Source' returns 100, so the outer query becomes:

SELECT * FROM employees WHERE department_id = 100;

6.2 Operators with Subqueries

Use = when the subquery returns exactly ONE value:

WHERE department_id = (SELECT id FROM departments WHERE name = 'Open Source')

Use IN when the subquery returns MULTIPLE values:

WHERE department_id IN (
    SELECT id FROM departments WHERE name = 'Open Source' OR name = 'Java'
)

⭐ What to Remember: Using = with a subquery that returns more than one row will throw an error. Always use IN when the subquery can return multiple rows.

Use >, <, >=, <= with a single-value subquery:

-- Employees in departments with higher ID than Open Source
WHERE department_id > (SELECT id FROM departments WHERE name = 'Open Source')

6.3 ANY and ALL Keywords

When comparing against a multi-value subquery with > or <:

ANY (equivalent to OR):

WHERE department_id > ANY (
    SELECT id FROM departments WHERE name = 'Open Source' OR name = 'Java'
)
-- Means: department_id > 100 OR department_id > 200
-- = department_id > 100 (the smaller value)

ALL (equivalent to AND):

WHERE department_id > ALL (
    SELECT id FROM departments WHERE name = 'Open Source' OR name = 'Java'
)
-- Means: department_id > 100 AND department_id > 200
-- = department_id > 200 (must be greater than the largest)
Keyword Meaning
= ANY Same as IN
> ANY Greater than the minimum value
> ALL Greater than the maximum value
< ANY Less than the maximum value
< ALL Less than the minimum value

Subquery Readability Issue:

-- Nested subqueries are read from inside → outside (hard to follow)
SELECT * FROM films
WHERE id IN (
    SELECT film_id FROM film_category
    WHERE category_id = (
        SELECT id FROM category WHERE name = 'Action'
    )
);

This is hard to read as complexity grows. Solution → CTEs.

7. Common Table Expressions (CTEs)

7.1 What Is a CTE?

A CTE (Common Table Expression) is a named temporary result set defined at the top of a query using the WITH keyword. It makes complex queries more readable by breaking them into named steps that you read top-to-bottom.

Key advantage: Readability — you can follow the logic like reading a document, step by step.

7.2 CTE Syntax 

WITH cte_name AS (
    -- Your query here
    SELECT ...
)
SELECT * FROM cte_name;

7.3 Chaining Multiple CTEs 

WITH
  action_category AS (
      SELECT id
      FROM category
      WHERE name = 'Action'
  ),
  action_film_ids AS (
      SELECT film_id
      FROM film_category
      WHERE category_id IN (SELECT id FROM action_category)
  )
SELECT *
FROM film
WHERE id IN (SELECT film_id FROM action_film_ids);

How it reads:

  1. action_category → get the ID of the "Action" category.
  2. action_film_ids → get all film IDs that belong to "Action".
  3. Final SELECT → retrieve full film details.

CTE Rules:

  • A CTE can only reference CTEs defined before it (above it in the WITH clause).
  • You cannot reference a later CTE from an earlier one.
  • The final SELECT after all CTEs is the main query.

7.4 CTE vs. Subquery

Feature Subquery CTE
Readability Read inside → out (harder) Read top → bottom (easier)
Reusability in query Must repeat if needed multiple times Define once, reference many times
Recursion Not supported Supported (Recursive CTE)
Scope Only within the surrounding query Throughout the entire query
In exam? Yes Mentioned as bonus knowledge

Note from Instructor: CTEs are not in the exam syllabus but are important in real-world work.

Recursive CTEs (advanced): A CTE can call itself, similar to recursion in programming. Must have a base condition to stop.

8. Views

8.1 Simple (Regular) View

A View is a stored, named SELECT query saved in the database. It acts like a virtual table — you can SELECT from it like a table, but the underlying data comes from real tables.

Key characteristics of a Simple View:

  • Does NOT store data on disk (no extra space).
  • Every time you query a simple view, it re-executes the underlying query against the real tables.
  • Persists across sessions (unlike CTEs which disappear after the query ends).
  • Used for: security (hide sensitive columns), reusability (avoid re-writing complex queries), abstraction (present a simpler interface to users).

8.2 Creating a View 

CREATE VIEW long_films AS
SELECT film_id, title, description, length
FROM film
WHERE length > 100;

Query the view:

SELECT * FROM long_films;
SELECT * FROM long_films WHERE film_id = 147;

8.3 Updatable Views — Rules

A view is updatable (you can INSERT/UPDATE through it) only when all of these conditions hold:

  1. No DISTINCT keyword.
  2. No aggregate functions (SUM, COUNT, AVG, etc.).
  3. No GROUP BY or HAVING.
  4. No UNION.
  5. No subqueries in the SELECT list.
  6. The view references exactly one base table (no JOINs).
  7. Must be a one-to-one relationship between view rows and table rows.

If any condition is violated → the view is NOT updatable.

Example — Non-updatable view (uses SUM and GROUP BY):

CREATE VIEW total_per_customer AS
SELECT customer_id, SUM(amount) AS total
FROM payment
GROUP BY customer_id;

-- This will fail:
UPDATE total_per_customer SET customer_id = 2 WHERE customer_id = 1;
-- Error: The target table is not updatable

Why? One row in the view (customer 1's total) represents multiple rows in the base table (multiple payments). MySQL cannot determine which base row to update.

Example — Updatable view (simple SELECT):

CREATE VIEW long_films AS
SELECT * FROM film WHERE length > 100;

-- This WORKS:
UPDATE long_films SET title = 'New Title' WHERE film_id = 147;
-- It's equivalent to: UPDATE film SET title = 'New Title' WHERE film_id = 147;

8.4 Simple View vs. Direct Query — Performance

Does using a simple view improve performance compared to running the query directly?

Answer: No — there is NO performance difference between a simple view and running the underlying query directly.

  • A simple view stores the query definition, not the results.
  • Every time you call the view, MySQL executes the query from scratch against the base tables.
  • It is identical in performance to writing the query yourself each time.

Why use a simple view then?

  • Reusability: Write the complex query once, reference by name.
  • Security: Expose only certain columns/rows to users.
  • Persistence: Unlike CTEs, views survive after the session ends.
  • Abstraction: Simplify complex joins for other developers.

8.5 CTE vs. View — Key Differences

Feature CTE Simple View
Scope Only within the single query it's defined in Persistent — available across sessions
Storage Not stored; vanishes after query ends Stored in database permanently
Performance Same as inline query Same as inline query
Reuse across queries No Yes
Creation syntax WITH name AS (...) CREATE VIEW name AS (...)

8.6 Materialized View

Note: Materialized Views are NOT available in MySQL. They exist in PostgreSQL and other database systems. This section was covered for general knowledge.

A Materialized View is a view that physically stores its query results on disk.

How it works:

  1. When created, the query executes and the results are saved to disk (like a real table).
  2. Subsequent queries read from the stored results — not from the base tables.
  3. To get updated results, you must REFRESH the materialized view manually.

Creating a Materialized View (PostgreSQL syntax):

CREATE MATERIALIZED VIEW long_films AS
SELECT film_id, title, description, length
FROM film
WHERE length > 100;

-- Refresh to get latest data:
REFRESH MATERIALIZED VIEW long_films;

Key behaviors:

  • Table changes DO NOT automatically sync to the materialized view — you must REFRESH.
  • You cannot UPDATE a materialized view directly — it's read-only.
  • Data may be stale until REFRESH is called.

8.7 Simple View vs. Materialized View — Comparison

Feature Simple View Materialized View
Data stored on disk ❌ No ✅ Yes
Query re-runs on SELECT ✅ Every time ❌ No — reads from disk
Performance (read) Same as base query Faster — pre-computed
Performance (write) N/A Overhead on REFRESH
Data freshness Always current Stale until REFRESH
Space usage None Takes disk space
Updatable? Sometimes (see rules) ❌ Read-only
Reflects table changes? ✅ Immediately ❌ Only after REFRESH
Available in MySQL? ✅ Yes ❌ No (PostgreSQL only)

When to use a Materialized View:

  • The underlying query is complex and expensive (e.g., joins, aggregations across millions of rows).
  • The data doesn't change frequently (or you can tolerate slightly stale data).
  • You need fast reads and the query result set is manageable in size.

Example: 1,000,000 order records. You need Q1 orders (100,000 records) frequently.

Without materialized view → scan 1,000,000 rows every time.
With materialized view → scan 100,000 pre-filtered rows, then apply any additional filters.

10. Potential Exam Questions

Conceptual Questions

  1. What is the main difference between TIMESTAMP and DATETIME in MySQL?
    → Timezone handling: TIMESTAMP auto-converts based on session timezone; DATETIME stores as-is.

  2. Why does TIMESTAMP have a maximum date of 2038?
    → It's stored as a 32-bit Unix integer, which overflows on January 19, 2038.

  3. What is the difference between CHAR and VARCHAR?
    → CHAR is fixed-length (always occupies n bytes); VARCHAR is variable-length (uses actual length + 1–2 bytes).

  4. What are the two implicit constraints in a PRIMARY KEY?
    → UNIQUE and NOT NULL.

  5. How does UNIQUE differ from PRIMARY KEY?
    → A table can have only one PRIMARY KEY, but multiple UNIQUE constraints. UNIQUE alone allows one NULL value.

  6. What does a FOREIGN KEY constraint enforce?
    → Referential integrity: any value in the child's FK column must exist in the parent's referenced column.

  7. What are the three referential actions for ON DELETE / ON UPDATE?
    → RESTRICT (default), CASCADE, SET NULL.

  8. What happens with ON DELETE CASCADE?
    → When a parent row is deleted, all child rows referencing it are automatically deleted.

  9. What happens with ON DELETE SET NULL?
    → When a parent row is deleted, the child rows' FK column is set to NULL (the child rows remain).

  10. What is a subquery and when do you use IN instead of =?
    → A query nested inside another query. Use = when the subquery returns exactly one value; use IN when it may return multiple values.

  11. What is the difference between ANY and ALL with comparison operators?
    → > ANY means greater than the minimum; > ALL means greater than the maximum.

  12. What is the main advantage of a CTE over a subquery?
    → Readability — CTEs are named, read top-to-bottom, and can be reused within the same query.

  13. What is a View, and does a simple view improve query performance?
    → A view is a stored named query. A simple view does NOT improve performance — it re-executes the underlying query every time.

  14. When is a view NOT updatable?
    → When it uses DISTINCT, aggregate functions, GROUP BY, HAVING, UNION, subqueries in SELECT, or JOINs across multiple tables.

  15. What is a Materialized View and how does it differ from a simple view?
    → A materialized view stores its results physically on disk. It's faster to read but can become stale (must REFRESH). A simple view stores only the query definition, re-executes every time, and always shows current data.

  16. Why might you prefer a NoSQL document database for a social profile use case?
    → Because all user data (profile, jobs, education) can be stored as one document, enabling fast single-read retrieval without expensive JOINs across multiple tables.

  17. What is the trade-off of document serialization in NoSQL?
    → Fast reads (all data together), but slow updates (must rewrite the entire document to change one field).

  18. What is the purpose of ORM?
    → To bridge the impedance mismatch between object-oriented backend code (classes/objects) and relational database storage (rows/columns).

  19. What is an ENUM and how is it stored internally?
    → A column type with a predefined list of allowed string values. Stored internally as integer indexes, not strings.

  20. What is the rule for expressions in a generated column?
    → The expression must be deterministic — the same input must always produce the same output. Functions like NOW() or RAND() are not allowed.

SQL Writing Questions

Q: Create a table employees with id (PK), name (NOT NULL), email (UNIQUE NOT NULL), salary (CHECK >= 30000), and department_id (FK to departments.id, CASCADE on delete).

CREATE TABLE departments (
  id   INT          PRIMARY KEY,
  name VARCHAR(100) NOT NULL
);

CREATE TABLE employees (
  id            INT         PRIMARY KEY,
  name          VARCHAR(30) NOT NULL,
  email         VARCHAR(100) UNIQUE NOT NULL,
  salary        INT         CHECK (salary >= 30000),
  department_id INT,
  CONSTRAINT fk_emp_dept
    FOREIGN KEY (department_id) REFERENCES departments(id)
    ON DELETE CASCADE
    ON UPDATE RESTRICT
);

Q: Write a subquery to find all employees in departments with "Open Source" or "Java" in their name.

SELECT *
FROM employees
WHERE department_id IN (
    SELECT id
    FROM departments
    WHERE name = 'Open Source' OR name = 'Java'
);

Q: Rewrite the above as a CTE.

WITH target_departments AS (
    SELECT id
    FROM departments
    WHERE name = 'Open Source' OR name = 'Java'
)
SELECT *
FROM employees
WHERE department_id IN (SELECT id FROM target_departments);

Q: Create a view showing all films with length > 100 minutes.

CREATE VIEW long_films AS
SELECT film_id, title, description, length
FROM film
WHERE length > 100;

Q: Create a table users with email and an auto-generated domain column.

CREATE TABLE users (
  email  VARCHAR(100),
  domain VARCHAR(100) AS (SUBSTRING_INDEX(email, '@', -1)) STORED
);
